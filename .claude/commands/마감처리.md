"마감처리"다. 매 단계 묻지 말고 **전 과정을 끝까지 자동 수행**한다. 단, 중간에 실패가 나면 거기서 멈추고 원인을 보고한다.

배포 메커니즘: 로컬 deploy 스크립트 없음 → **origin master 푸시 시 GitHub 연동 호스팅 자동 배포**. 원격: `ihaelyong988-lab/zeros-estimate-service`.

## 절차
1. **그래프 최신화(코드 변경 시)**: `.ts/.tsx`가 변경됐으면 `graphify update .` 1회.
2. **빌드 검증**: `npm run build` → **에러 0건** 확인. 실패 시 **즉시 중단·원인 보고**(커밋/푸시 금지).
3. **브랜치**: 현재 `master`면 변경 내용을 대표하는 새 브랜치 생성(예: `feat/...`, `fix/...`, `refactor/ui-...`). 이미 작업 브랜치면 그대로 사용.
4. **작업 기록 자동 갱신 (필수 — 사용자가 시키지 않아도 매번 자동)**:
   - `docs/_worklog/<오늘날짜 YYYY-MM-DD>_작업정리.md`에 이번 작업을 **항목별로 추가**(같은 날짜 파일이 있으면 새 섹션으로 이어붙이고, 없으면 생성). 무엇을·왜·어디(파일/위치)·검증법을 남긴다.
   - `docs/00_orchestration/state.md`의 **"최신 반영"** 에 한 줄 추가(날짜·요약·커밋해시 자리) + **"다음 할 일"** 갱신.
   - 이 문서 변경을 **같은 커밋에 포함**한다(코드+기록 1커밋). 커밋 후 해시를 state/worklog에 채워도 됨.
5. **커밋**: 변경을 의미 단위로 스테이징하고 한국어 요약 메시지로 커밋. $1 가 주어지면 그 문구를 제목에 반영. 메시지 끝에 반드시:
   ```
   Co-Authored-By: Claude Opus 4.8 <noreply@anthropic.com>
   ```
6. **브랜치 푸시**: `git push -u origin <branch>`.
7. **master 머지**: `master` 체크아웃 → 작업 브랜치 머지(가능하면 `--no-ff`).
8. **master 푸시**: `git push origin master` → **호스팅 자동 배포 트리거**.
9. **배포 후 프로브(필수)**: `node scripts/prod-probe.mjs` — 사이트 200 · OTP 문자 활성 여부 · 관리자 로그인 env 3종을 라이브에 실측. WARN은 마감을 차단하지 않지만 **마무리 보고에 결과를 반드시 포함**한다(키 미등록 상태를 "정상"으로 보고 금지).
10. **마무리 보고**: 빌드 결과·브랜치명·커밋 해시·푸시 결과·프로브 결과를 표/짧은 목록으로 보고. 배포 반영은 호스팅측 처리이므로 "푸시 완료=배포 트리거됨"으로 명시.

## 원칙
- 단정적/과장 표현 금지. 사실만 보고(실패는 실패라고).
- 사용자가 의도적으로 지운/단순화한 것을 임의로 되살리지 않는다.
- `master` 직접 커밋 금지 — 항상 브랜치 경유.
